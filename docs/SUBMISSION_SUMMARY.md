# 베프 관리자 웹 제출 요약

## 한 줄 설명

베프 관리자 웹은 장애학생의 접근성 제보, 도움 요청, 경험 피드를 학교 개선 업무와 월간 리포트로 연결하는 공모전용 운영 대시보드입니다. 공공데이터와 추천 기능은 최종 판단이 아니라 관리자 검토를 돕는 보조 근거로 사용합니다.

## 구현 범위

| 구분 | 상태 |
|---|---|
| 관리자 로그인/내 정보 | Spring Boot API 연결 |
| 대시보드 KPI | Spring Boot API 연결 |
| 접근성 제보 | 목록/상태/담당자/우선순위/메모/첨부파일 API 연결 |
| 도움 요청 | 목록/상태/센터 판단 API 연결 |
| 경험 피드 | 목록/공개 상태/규칙 기반 익명화 API 연결 |
| 개선 워크플로우 | 목록/단계 변경 API 연결 |
| 월간 리포트 | 조회/PDF/CSV/스냅샷 API 연결 |
| 공공데이터 | seed/동기화/원본/정규화/지도 레이어 표시 |
| 감사 로그 | 조회/CSV API 연결 |
| 실제 외부 LLM | 추가 예정 |
| 운영 DB/Oracle | 승인 후 별도 진행 |

## 시연 핵심 흐름

1. `/#/dashboard`: 반복 제보와 도움 요청 현황
2. `/#/reports`: 제보 처리, 담당자 배정, 첨부파일
3. `/#/public-data`: 공공데이터 수집 결과, 원본 JSON, 정규화, 지도 레이어의 한계 확인
4. `/#/analysis`: 반복 문제 분석과 관리자 검토용 추천
5. `/#/monthly-report`: PDF/CSV 리포트 내보내기

## 정확히 말해야 할 공공데이터 상태

- 공공데이터 후보: 70종
- 백엔드 동기화 슬롯: 68개
- 현재 실제 호출 검증: 3개
- endpoint 미설정: 65개
- 확인되지 않은 endpoint는 실제 연동이라고 말하지 않습니다.
- 공공데이터는 캠퍼스 현장 상태를 완전히 설명하기 어렵기 때문에, 계획서에서는 핵심 기능보다 보조 검증 자료로 설명합니다.
- 좌표나 필드가 부족한 데이터는 실제 위치/상태로 단정하지 않고 후보 레이어로만 다룹니다.

## 정확히 말해야 할 AI/익명화 상태

- 우선 조치 추천은 AI가 업무를 결정하는 기능이 아니라, 위험도·반복성·공감·공공데이터 차이를 함께 보여주는 관리자 검토 보조 기능입니다.
- 고위험 제보는 공감 수나 반복 제보 수가 낮아도 우선 검토 대상에서 제외하지 않는 방향이 맞습니다. 계획서에서는 `위험도 우선, 공감/반복은 보조 근거`라고 설명합니다.
- 익명화는 모든 제보의 편향 제거 목적이 아니라, 장애학생이 소수라 특정 수업·건물·상황만으로 개인이 식별될 수 있는 경우를 막기 위한 보호 장치입니다.
- 실제 외부 LLM은 아직 연결하지 않았고, 현재는 규칙 기반 MVP입니다.

## 배포 주의

- Dothome은 정적 프론트만 가능합니다.
- Spring Boot 백엔드는 별도 Java 서버가 필요합니다.
- 공개 배포에서 HTTP 모드를 쓰려면 `VITE_API_BASE_URL`을 별도 백엔드 서버 주소로 설정해야 합니다.
