# 베프 관리자 웹 발표 시연 스크립트

## 핵심 메시지

베프 관리자 웹은 장애학생의 개별 불편을 단순 민원으로 끝내지 않고, 위험도와 현장 제보를 중심으로 학교 개선 근거를 정리하는 운영 대시보드입니다. 반복 제보와 공감은 중요한 보조 근거지만, 고위험 제보를 배제하는 기준으로 쓰지 않습니다.

## 3분 시연 흐름

| 순서 | 화면 | 말할 내용 |
|---:|---|---|
| 1 | `#/dashboard` | "오늘 학교가 먼저 봐야 할 접근성 문제를 KPI와 관리자 검토용 추천 카드로 확인합니다." |
| 2 | `#/public-data` | "공공데이터는 참고 레이어이고, 실제 이용 가능성은 현장 제보와 관리자 확인이 필요합니다." |
| 3 | `#/reports` | "제보 상세에서 담당자, 우선순위, 처리 메모, 개선 요청서 초안을 확인합니다." |
| 4 | `#/help-requests` | "긴급 도움 요청은 단순 호출 로그가 아니라 이후 시설 개선 제보로 연결될 수 있습니다." |
| 5 | `#/stories` | "장애학생이 소수라 특정될 수 있는 표현은 익명화와 관리자 검수를 거쳐 공개합니다." |
| 6 | `#/monthly-report` | "월간 리포트는 학교와 시설팀에 제출할 수 있는 개선 근거 자료가 됩니다." |

## 5분 시연 흐름

1. `#/dashboard`
   - 전체 제보 수, 공감 수, 미해결 건, 도움 요청 수를 보여줍니다.
   - "관리자는 위험도를 먼저 보고, 반복 제보와 공감 수는 우선 검토를 돕는 보조 근거로 함께 봅니다."

2. `#/public-data`
   - 공공데이터와 실제 제보 사이의 차이를 보여줍니다.
   - "공공데이터는 참고 레이어이고, 베프의 핵심은 현장 제보와 관리자 확인입니다."

3. `#/reports`
   - 검색바로 특정 건물을 찾습니다.
   - 담당자 배정과 우선순위 변경을 보여줍니다.
   - "Mock 모드에서는 새로고침하면 seed 상태로 돌아가고, HTTP 모드에서는 담당자/우선순위/메모가 Spring Boot API에 저장됩니다."

4. `#/help-requests`
   - 도움 요청 상세 패널을 보여줍니다.
   - 센터 판단, 처리 이력, 연결 후보 제보를 설명합니다.

5. `#/stories`
   - 익명화 전/후 비교를 보여줍니다.
   - "익명화는 편향 제거가 아니라, 소수자인 장애학생이 수업·건물·상황만으로 특정될 위험을 줄이기 위한 보호 장치입니다."

6. `#/workflow`
   - 제보가 개선 과제로 넘어가는 단계를 보여줍니다.
   - "개별 제보가 시설 개선 업무로 이어지는 흐름입니다."

7. `#/monthly-report`
   - 건물별 리포트, 고위험 미해결 항목, 개선 요청서 미리보기를 보여줍니다.
   - "학교가 예산과 시설 개선 우선순위를 정할 수 있는 자료로 전환됩니다."

8. `#/settings`
   - Mock/API 상태, 권한 정책, 백엔드 체크리스트를 보여줍니다.
   - "프론트는 시연용으로 완성했고, 실제 서비스화를 위해 필요한 백엔드 작업을 분리했습니다."

## 발표 때 조심할 표현

- "전부 운영 배포됐습니다"라고 말하지 않습니다.
- "관리자 웹 주요 기능은 Spring Boot API에 연결했고, 운영 DB/외부 LLM/기관 자동 전송은 추가 예정으로 분리했습니다"라고 설명합니다.
- "AI가 결정합니다"보다 "추천 기능은 관리자 검토를 돕는 초안 기능입니다"라고 설명합니다.
- "AI가 관리자 행동까지 결정합니다"라고 말하지 않고, "선택지를 넓히고 근거를 정리합니다"라고 설명합니다.
- "공공데이터가 완벽합니다"보다 "공공데이터는 참고 레이어이며 현장 제보와 관리자 확인이 더 중요합니다"라고 설명합니다.
- "공감과 반복이 낮으면 제외합니다"라고 말하지 않고, "고위험 제보는 공감/반복이 낮아도 먼저 확인합니다"라고 설명합니다.
